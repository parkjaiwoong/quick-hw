# 고객 연결 → 기사 배송완료 → 출금 요청 → 관리자 승인 → 기사 출금 프로세스

---

## 관리자 화면: 정산 관리 vs 출금(지급) 관리

| 구분 | 정산 관리 (`/admin/settlements`) | 출금 관리 (`/admin/payouts`) |
|------|----------------------------------|------------------------------|
| **역할** | 배송 건별 **정산 내역** 확인·관리. “얼마를 벌었는지” 단위. | 기사가 **요청한 출금**을 승인·이체하는 단위. |
| **대상** | **settlements** (배송 1건 = 정산 1건) | **payout_requests** (기사가 “이만큼 출금해 주세요” 요청한 건) |
| **주요 액션** | **정산 확정** (결제완료 건을 “출금 가능”으로 전환), 엑셀 다운로드, 잠금 해제 | **승인** → **이체 완료** (또는 보류/반려) |
| **언제 쓰나** | 정산 금액·상태를 보고, 필요 시 **정산 확정**으로 기사 지갑의 “출금 가능 금액”을 채울 때 | 기사가 출금 요청한 뒤, **승인**하고 실제 이체 후 **이체 완료** 처리할 때 |

- **정산 관리**: 기사별·배송 건별 **정산 목록**이 보이고, “정산확정 (N건)” 버튼으로 **결제 완료된 정산**을 일괄 확정한다. 확정하면 해당 금액이 기사 지갑의 **출금 가능 잔액(available_balance)**으로 넘어간다.
- **출금 관리**: 기사가 **지갑에서 출금 요청한 건**만 보인다. 여기서 **승인** → (실제 계좌 이체 후) **이체 완료**를 한다. 정산 목록을 보는 화면이 아니다.

---

## 정산 확정은 언제 하나? (시점 정리)

**정산 확정**은 **관리자 > 정산 관리** 화면에서만 한다. “결제는 이미 완료(PAID)인데, 아직 출금 가능 잔액으로 안 넘어간 정산”을 **출금 가능 상태로 넘기는** 단계다.

1. **배송 완료 시점 (자동)**  
   - 정산 1건 생성 + 지갑 **pending** 증가.  
   - 결제가 이미 **PAID** (카드/계좌이체 완료)이면:  
     - 정산은 **READY**, 지갑은 **pending → available** 까지 자동 반영.  
     - → **이 경우 정산 확정을 하지 않아도** 기사는 출금 요청 가능.
2. **결제가 PAID가 아닐 때 (예: 현금·나중에 결제 완료)**  
   - 정산은 **PENDING**, 지갑은 **pending**만 있고 available은 그대로.  
   - → 관리자가 **정산 관리**에서 해당 건(들)을 **정산 확정**하면:  
     - 정산 `settlement_status = CONFIRMED`, `move_driver_wallet_pending_to_available` 호출 → **available** 증가.  
     - 이후 기사가 출금 요청 가능.

**요약**:  
- **카드/계좌이체**처럼 배송 완료 시점에 이미 PAID면 → 정산 확정 **없이** 출금 요청 가능.  
- **현금** 등 나중에 PAID 처리하거나, 수동으로 “이 정산을 출금 가능으로 넘기고 싶을 때” **정산 관리**에서 **정산 확정**을 누른다.

## 전체 흐름 요약

1. **고객** 배송 요청(기사 연결 요청)
2. **기사** 수락 후 배송 진행 → **배송 완료** 처리
3. **정산** 1건 생성(결제 완료 시 출금 가능 금액 반영)
4. **(선택)** 관리자 **정산 관리**에서 **정산 확정** → PENDING/READY 건을 출금 가능(available)으로 전환
5. **기사** 지갑에서 **출금 요청**
6. **관리자** **출금 관리**에서 **승인**
7. **관리자** 실제 이체 후 **출금 관리**에서 **이체 완료** 처리 → 기사 계좌 출금 완료

---

## 단계별 상세

### 1. 고객 배송 요청(기사 연결)

- 고객이 배송 요청 시 `deliveries` 행 생성, 결제(카드/계좌이체/현금) 진행.
- 결제 완료 후 근처 기사에게 알림 → 기사가 수락하면 `deliveries.driver_id`·`accepted_at` 설정.

### 2. 기사 배송 완료

- 기사가 앱에서 **배송 완료** 처리 시:
  - `deliveries.status = 'delivered'`, `delivered_at` 저장.
  - **Server** `updateDeliveryStatus` → `createSettlementForDelivery`(after) 호출.

### 3. 정산 생성 및 지갑 반영

**`createSettlementForDelivery(deliveryId)`** (배송 완료 후 after()에서 실행):

- 해당 배송에 대한 **settlements** 1건 INSERT.
  - `settlement_status`: 결제가 이미 **PAID**이면 `READY`, 아니면 `PENDING`.
  - `payment_status`: 결제 상태(PAID 등).
- **driver_wallet**:
  - `increment_driver_wallet_pending(driver_id, 금액)` → `pending_balance` 증가.
  - 결제가 이미 PAID이면 `move_driver_wallet_pending_to_available` → **pending → available** 이전.
- 현금 결제인 경우: 결제·정산을 PAID로 갱신하지만, **available**로 옮기려면 관리자 **정산 확정**이 필요할 수 있음(구현에 따라 다름).

**관리자 정산 확정(선택):**

- **관리자 > 정산 관리**에서 **정산 확정** 시:
  - `confirmSettlement(settlementId)` → 해당 정산 `settlement_status = 'CONFIRMED'`.
  - `move_driver_wallet_pending_to_available` 호출 → **출금 가능 잔액(available_balance)** 증가.
- 카드/계좌이체로 이미 PAID인 건은 배송 완료 시점에 READY + available 반영되므로, 정산 확정 없이도 출금 요청 가능.

### 4. 기사 출금 요청

- **기사 > 지갑/출금** 화면에서만 **출금 요청** (출금 계좌는 미리 설정, 금액만 입력).
- **`requestPayout(driverId, amount)`**:
  - LOCKED 정산 있으면 출금 요청 불가.
  - **READY** 또는 **CONFIRMED** 이고 **payment_status = PAID**인 정산이 1건 이상 있어야 함.
  - 이미 **requested / on_hold / approved** 상태인 출금 요청이 있으면 추가 요청 불가.
  - `amount`는 `min_payout_amount` 이상, `available_balance` 이하.
  - **payout_requests** INSERT (status: `requested`), **driver_wallet.available_balance**에서 요청 금액 차감.

### 5. 관리자 출금 승인

- **관리자 > 출금 관리**에서 해당 출금 요청 **승인**.
- **`approvePayout(payoutId)`**:
  - 해당 기사의 **READY(결제완료)** 또는 **CONFIRMED(정산확정)** 이면서 아직 다른 출금에 묶이지 않은 정산들로 요청 금액을 채움.
  - 채울 수 있으면:
    - **payout_requests**: `status = 'approved'`, `payout_status = 'WAITING'`, `approved_at` 등 설정.
    - 해당 **settlements**: `settlement_status = 'CONFIRMED'`, `payout_request_id = payoutId`로 연결.

### 6. 관리자 이체 완료(기사에게 실제 출금)

- **관리자 > 출금 관리**에서 **이체 완료 처리** 클릭.
- **`transferPayout(payoutId, 'MANUAL')`**:
  - **payout_requests**: `status = 'transferred'`, `transferred_at`, `payout_status = 'PAID_OUT'` 등 설정.
  - 해당 **payout_request_id**로 연결된 **settlements**: `settlement_status = 'PAID_OUT'`, `status = 'completed'`.
  - 실제 계좌 이체는 관리자가 별도(뱅킹/PG 등)로 수행한 뒤, 이 버튼으로 “이체 완료”만 기록.

→ 여기까지 완료되면 **기사에게 출금된 것**으로 시스템상 프로세스 종료.

## 출금 보류와 보류 이후 처리

### 보류 vs 반려: 어떤 경우에 쓰나?

| 구분 | 의미 | 사용 예시 |
|------|------|-----------|
| **보류** | 당장 승인/반려하지 않고 **검토만 대기**할 때. 나중에 **이 요청 그대로 승인**할 수 있는 경우. | 내부 서류 검토 중, 특정 건만 추가 확인 후 곧 승인 예정, 당일 이체 일정 조율 중 |
| **반려** | 이 출금 요청을 **취소**하고, 금액을 기사 지갑으로 되돌릴 때. 기사가 **뭔가 수정한 뒤 다시 출금 요청**해야 하는 경우. | **계좌번호/은행 정보 오류**, 서류 불비, 출금 불가 사유로 같은 요청으로는 승인 불가한 경우 |

- **계좌번호가 틀렸다면 → 반려**가 맞습니다.  
  사유에 "계좌번호가 일치하지 않습니다. 올바른 계좌로 수정 후 다시 출금 요청해 주세요" 등을 입력하고 **반려**하면, 해당 금액이 기사 **출금 가능 잔액**에 다시 더해지고, 기사가 계좌 정보 수정 후 **새 출금 요청**을 하면 됩니다.  
- **보류**는 "이 요청은 유효한데, 당장만 미루는 것"에 가깝고, **반려**는 "이 요청은 받아들일 수 없으니 취소하고 다시 요청하세요"에 가깝습니다.

### 보류란?

- 관리자가 출금 요청을 **당장 승인/반려하지 않고** 검토할 때 사용합니다.
- **관리자 > 출금 관리**에서 해당 요청에 **보류** 클릭 → 사유 입력 → **`holdPayout(payoutId, reason)`** 실행.
- **payout_requests**: `status = 'on_hold'`, `notes = 사유`, `settlement_status = 'HOLD'`, `settlement_locked = true`.
- 기사 지갑의 **available_balance**는 출금 요청 시점에 이미 차감된 상태이므로, 해당 금액은 이 출금 요청에 묶인 상태로 유지됩니다. (반려 시에만 available 복구)

### 보류 후 기사가 할 작업

- **없습니다.** 보류는 "관리자 검토 대기" 상태일 뿐이며, 기사가 별도로 수정·제출할 것은 없습니다.
- 기사 앱 **지갑/출금 내역**에서는 해당 건이 **보류**로 표시되고, 관리자가 승인하거나 반려할 때까지 대기합니다.

### 보류 이후 관리자가 할 수 있는 것 (재승인 프로세스)

1. **승인**  
   - **관리자 > 출금 관리**에서 보류된 건을 선택 후 **「보류 해제 후 승인」** 클릭.  
   - **`approvePayout(payoutId)`**가 호출되며, **보류(on_hold) 건이면** 먼저 해당 출금 요청을 `status = requested`, `settlement_status = READY`, `settlement_locked = false` 로 되돌린 뒤, 기존과 동일한 승인 로직(정산 할당 → approved)을 수행합니다.  
   - 따라서 **기사가 따로 할 작업 없이** 관리자만 "보류 해제 후 승인"을 누르면 출금 승인까지 이어집니다.

2. **반려**  
   - 보류된 건에 대해 **반려**를 선택하고 사유 입력 → **`rejectPayout(payoutId, reason)`**.  
   - 출금 요청이 `rejected` 처리되고, 해당 금액이 기사 **available_balance**에 다시 더해지며, 기사는 이후 같은 금액으로 다시 출금 요청할 수 있습니다.

정리하면, **출금 보류 이후**에는  
- 기사: 별도 작업 없음.  
- 관리자: **「보류 해제 후 승인」**으로 재승인하거나, **반려**로 출금 요청을 취소하고 금액을 기사 지갑에 되돌리는 두 가지 선택만 있으면 됩니다.

## 데이터/상태 정리

| 구분 | 테이블/필드 | 설명 |
|------|-------------|------|
| 배송 | deliveries.status | pending → accepted → picked_up / in_transit → delivered |
| 정산 | settlements | delivery_id별 1건, settlement_status: PENDING → READY/CONFIRMED → PAID_OUT |
| 지갑 | driver_wallet | pending_balance, available_balance (출금 요청 시 available 차감) |
| 출금 요청 | payout_requests | status: requested → approved → transferred (또는 on_hold → 승인/반려, rejected 등) |

---

## 문제점 및 수정 사항

- **approvePayout**: 이전에는 “CONFIRMED 정산 금액”만 검사하고, 실제 할당은 “READY 정산”만 사용해,  
  정산 확정을 하지 않으면 CONFIRMED가 없어 승인 실패하거나, 정산 확정만 하면 READY가 없어 할당 실패하는 모순이 있었음.  
  → **수정**: 출금 할당 대상을 **READY(결제완료) + CONFIRMED(정산확정)** 이면서 `payout_request_id`가 null인 정산으로 통합하고,  
  이 풀에서 요청 금액만큼 순서대로 할당하도록 변경함.  
  따라서 “정산 확정을 하지 않고 결제만 완료(READY)”인 경우에도 출금 승인·이체 완료까지 진행 가능.

## 최종: 기사에게 출금되기까지

1. 고객 배송 요청 → 기사 수락 → 기사 배송 완료.
2. 배송 완료 시 정산 1건 생성, 결제 완료 건은 READY + 지갑 available 반영(또는 관리자 정산 확정으로 CONFIRMED + available 반영).
3. 기사가 지갑 화면에서 출금 요청 → `payout_requests` 생성, available 차감.
4. 관리자가 출금 관리에서 **승인** → 해당 정산들과 출금 요청 연결.
5. 관리자가 실제 계좌 이체 후 **이체 완료 처리** → 출금 요청·정산 상태가 이체 완료(PAID_OUT)로 갱신.

이 순서가 모두 끝나면 “기사에게 출금된 것”까지의 프로세스가 완료된 상태입니다.

## 일반적인 출금 프로세스와 비교

| 단계 | 일반적인 마켓플레이스/플랫폼 | 본 서비스 (현재 구현) |
|------|-----------------------------|---------------------------|
| 1. 수익 발생 | 주문/배송 완료 시 수익 확정 | 배송 완료 시 정산 1건 생성, 지갑 **pending** 증가 |
| 2. 출금 가능 전환 | 정산 주기(일/주/월)에 따라 “정산 완료” 후 출금 가능 잔액으로 이동 | **결제 PAID**면 자동으로 **available** 반영. 현금 등은 관리자 **정산 확정** 시 available 반영 |
| 3. 출금 요청 | 크리에이터/판매자가 “출금 요청” 버튼으로 금액 입력 후 요청 | 기사가 **지갑**에서 출금 계좌 설정 후 금액만 입력해 출금 요청 |
| 4. 승인/검토 | 자동 승인 또는 관리자 승인 | **관리자 출금 관리**에서 **승인** (보류/반려 가능) |
| 5. 실제 지급 | PG/뱅킹으로 자동 이체 또는 수동 이체 후 “지급 완료” 처리 | 관리자가 실제 계좌 이체 후 **이체 완료 처리**로 시스템에 반영 |

**차이점 요약**

- **정산 vs 출금 화면 분리**: 많은 플랫폼도 “정산(매출/정산서)” 화면과 “출금 요청/지급” 화면을 나눈다. 본 서비스도 **정산 관리** = 정산 내역·확정, **출금 관리** = 출금 요청 승인·이체 로 분리되어 있어 일반적인 구조와 유사하다.
- **정산 확정의 의미**: 카드/계좌이체는 배송 완료 시점에 이미 “결제 완료”이므로, 별도 **정산 확정** 없이 출금 가능 금액에 반영된다. **정산 확정**은 “결제는 완료됐지만 아직 출금 가능으로 안 넘긴 건”(예: 현금 수금 후 수동 반영)을 **출금 가능(available)**으로 넘기는 **선택 단계**다. 일반 플랫폼의 “정산 배치 완료” 또는 “수동 정산 반영”에 해당한다.
- **이체 완료는 수동**: 실제 돈 이체는 관리자가 뱅킹 등으로 수행한 뒤, **이체 완료 처리** 버튼으로 시스템에만 기록한다. 자동 이체(PG/뱅킹 API 연동)를 쓰지 않는 점은 많은 소규모 서비스와 동일한 방식이다.

## 현금 결제 시 처리 (기사가 이미 현금을 받은 경우)

### 현재 구현의 동작

- **배송 완료 시**: 현금 건도 **정산 1건 생성** (결제는 PENDING/READY → 배송 완료 시 PAID로 갱신). 정산은 **PENDING**, 지갑은 **pending**만 증가하고 **available**로는 안 넘어감.
- **관리자 정산 확정 시**: 해당 현금 정산을 **CONFIRMED**로 바꾸고 **pending → available** 이전. 그래서 기사가 그 금액을 **출금 요청**할 수 있게 됨.
- **기사 출금 요청 → 관리자 승인 → 이체 완료**: 현금 건도 카드와 동일한 흐름으로 **승인** 후 **이체 완료**까지 처리하게 됨.

즉, **“이미 기사가 현금을 받았는데도”** 시스템 상으로는  
정산 생성 → 정산 확정 → 출금 요청 → 승인 → **이체 완료**  
까지 가는 구조다.  
실제로는 현금 건은 **플랫폼이 기사에게 다시 송금하는 게 아니라**, 고객이 기사에게 직접 건넨 금액이므로 **“이체 완료”는 회계/기록용**으로만 쓰이게 된다.

### 왜 이렇게 되어 있나

- 모든 배송 건을 **동일한 정산·출금 플로우**로 쌓고, “정산 확정 → 출금 요청 → 승인 → 이체 완료”로 **통일된 기록**을 남기려는 방식이다.
- 현금 건은 **실제 계좌 이체는 하지 않아도** 되지만, 시스템에서는 “해당 정산까지 출금 처리 완료됐다”는 의미로 **이체 완료**를 눌러 두는 식으로 운영하는 경우가 있다 (실제 송금 없이 “수령·정산 완료”만 반영).

### 일반적으로 쓰는 방식

| 방식 | 설명 |
|------|------|
| **A. 현금 = 출금 대상에서 제외** | 현금 건은 **정산·매출 기록만** 하고, **출금 요청/이체 풀에는 넣지 않음**. 기사는 “이미 현금으로 받았음”으로 간주. 출금 요청 가능 금액은 **카드/계좌이체 건만** 모아서 계산. **이체 완료**는 “실제로 플랫폼이 송금한 건”만 사용. |
| **B. 현금 = 기사가 플랫폼에 납부 후 정산** | 기사가 현금을 **플랫폼/가맹점에 납부**하면, 그때부터 “플랫폼이 가진 돈”으로 간주하고 카드와 동일하게 **출금 가능 → 출금 요청 → 승인 → 이체** 처리. (택배/배달 가맹점 정산에서 자주 쓰는 방식) |
| **C. 현재처럼 통일 플로우 + 이체 완료는 기록용** | 카드/현금 모두 **같은 정산·출금 플로우**를 타고, 현금 건도 “출금 요청 → 승인 → 이체 완료”까지 **회계·내부 기록**으로만 진행. 실제 송금은 카드 건만 하고, 현금 건은 “이미 기사가 수령했으므로 이체 완료만 체크”하는 운영. |

### 정리

- **질문하신 대로**, 지금 구조에서는  
  - 현금을 기사가 이미 받아도 **정산은 하고**,  
  - 관리자가 **정산 확정**을 하면 기사가 그만큼 **출금 요청**할 수 있고,  
  - 승인 후 **이체를 실제로 하지 않아도** “기록상 출금 처리 완료”를 위해 **이체 완료**를 눌러 두는 식으로 갈 수 있다.
- **일반적인 프로세스**는  
  - **A**처럼 현금은 “수령 완료”로만 두고 **출금/이체 대상에서 빼거나**,  
  - **B**처럼 현금을 먼저 플랫폼에 납부한 뒤에만 출금 가능으로 넣는 방식이 많다.  
  **C**는 “모든 건을 같은 플로우로 기록하고, 현금은 이체 완료만 체크”하는 운영 방식에 가깝다.

**구현 반영 (현금 건 제외)**: `requestPayout`·`approvePayout`에서 payments.payment_method = 'cash' 인 정산은 출금 가능 합계/할당 풀에서 제외했다. 현금 건은 정산·지갑에는 올라가지만 출금 요청 가능 금액과 이체 대상에서는 제외된다.

(참고) 추후 **현금 수령 완료** 상태를 두어 이체 완료 없이 정산만 닫는 식으로 바꾸려면  
`requestPayout` / `approvePayout` 단계에서 **결제 수단(현금)** 이나 **정산 타입**을 보고 “출금 가능 금액/할당 풀”에서 제외하는 로직을 넣으면 된다.
